148  /*! \brief returns the closest distance of the plot element to the (screen pixel) position \a pos, or \c NAN
149 
150  This function is used to implement hit tests, i.e. to test whether a graph is close to a given position \a posSystem.
151  The function will then return the distance of the closes graph-point and a label for this point. An example of what
152  can be done with this function is the tooltip tool that JKQTPlotter provides via its context-menu/toolbar. This tool
153  uses just the information of the closest point and its label to display a tooltip for that datapoint:
154 
155  \see jkqtpmdaToolTipForClosestDataPoint for details.
156 
157 
158  \param posSystem position to test in system coordinates
159  \param[out] closestSpotSystem optional output of the closest point found on the plot element in system coordinates
160  \param[out] label optional output of a label for the closest point (that might e.g. be used in a tooltip)
161  \param mode search mode, i.e. use sqrt(dx*dx+dy*dy) as distance, or just the absoulte values along one of the
162  two coordinate axes. Note that the returned distance depends on this parameter!
163  \return NAN if not implemented, or if \a pos is very far from the plot element, or the closest distance (in screen pixels)
164  of \a pos from the plot element. Note that the returned distance depends on the choosen \a mode !!!
165  You can use JKQTPIsOKFloat() to check whether a valid distance was returned!
166 
167 
168  Since tha graph base class does not have any knowledge about how to perform a hit test on you specific graph, there is only a
169  very general implementation in this class, which does not actually search through the graph itself, but searches through
170  extra data that hs to be written during draw() and is stored in m_hitTestData. The implentation this base-class only searches this
171  list of points+metadata to implement a basic hit-test. If the list is empty, of no close-by points are found (default), then
172  hitTest() will simply return \a NAN.
173 
174  When writing a new graph, you can therefore implement hitTest() in one of these ways:
175  # You simply fill m_hitTestData with appropriate data and rely on the implementation in JKQTPPlotElement to do the work for you:
176  You then need to call clearHitTestData() at the start of your draw() function and whenever you draw a datapoint, you add
177  its location and metadata to the internal storage with addHitTestData()
178  # You derive from a graph class that already has an implementation. JKQTPXYGraph is an example of this. That class searches
179  through all x-/y-coordinates in the internally known columns and even takes into account possible graph errors in the label,
180  when the graph is also derived from JKQTPXGraphErrorData or JKQTPYGraphErrorData. This implementation therefore covers
181  most graph types pre-packaged with JKQTPlotter
182  # You implement the function from scratch
183  .
184 
185  \see addHitTestData(), clearHitTestData(), m_hitTestData, HitTestLocation
186 
187  */
188  virtual double hitTest(const QPointF & posSystem, QPointF* closestSpotSystem=nullptr, QString* label=nullptr, HitTestMode mode=HitTestXY) const;
